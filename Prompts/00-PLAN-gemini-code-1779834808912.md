# Plán implementace: AI Agents (AGENTS.MD)

Tento dokument definuje architekturu, orchestraci a chování LLM agentů v rámci aukční simulace. Cílem je vytvořit reaktivní, asynchronní systém, kde agenti simulují lidské ekonomické chování v reálném čase.

## 1. Architektura orchestrace agentů

Pro řízení komplexního rozhodovacího procesu a udržování stavu jednotlivých biderů využijeme **LangGraph** (případně v kombinaci se základním **LangChain** toolingem). 

* **State Management:** Každý agent bude v LangGraph reprezentován vlastním stavovým automatem.
    * `Global State`: Objekt předávaný enginem (aktuální cena, zbývající čas, historie posledních 5 bidů).
    * `Private State`: Interní paměť agenta (zbývající rozpočet, úroveň frustrace, skrytá maximální ochota platit - WTP).
* **Životní cyklus (Agent Loop):**
    1.  **Observe:** Přijetí nového ticku/události z aukčního enginu.
    2.  **Evaluate:** Zhodnocení stavu vůči Private State (Mám na to ještě? Dává to smysl?).
    3.  **Think (LLM Call):** Generování rozhodnutí na základě aktuálního promptu.
    4.  **Act:** Odeslání akce (Bid / Pass / Wait) zpět do REST API / event busu.

## 2. LLM Infrastruktura a Modely

Vzhledem k požadavkům na latenci (zejména u Penny aukcí, kde stovky agentů reagují na milisekundové změny) a finanční náročnosti běhu stovek souběžných API volání, je nezbytná lokální inference.

* **Inference Engine:** Pro dosažení maximální propustnosti nasadíme modely primárně přes **vLLM** nebo **NVIDIA NIM**, pro fázi prototypování případně **Ollama**.
* **Cílové modely:** Pro komplexní ekonomické uvažování s nízkou latencí využijeme modely v kategorii 8B–70B (např. **Llama 4**, **Qwen** příslušných kvantizací).
* **Integrace:** Aukční engine (ideálně napsaný v Go nebo Python/FastAPI pro asynchronní zátěž) bude komunikovat s inferenčními endpointy přes standardní OpenAI-compatible API.

## 3. Definice Rolí (Personas)

Při inicializaci simulace (`/api/bidders`) platforma vygeneruje pool agentů na základě následujících profilů. Každý profil má specifický `System Prompt` upravující váhy rozhodování.

### A. Opatrný Analytik (The Cautious)
* **Charakteristika:** Striktně racionální. Zná přesnou hodnotu předmětu a nikdy nepřihazuje nad stanovený limit.
* **Prompting pravidlo:** Jsi vysoce analytický kupec. Tvé rozhodnutí musí maximalizovat tvůj zisk (hodnota zboží - zaplacená cena). Nikdy nepřekračuj svůj budget.

### B. Gambler / FOMO (The Thrill-Seeker)
* **Charakteristika:** Vysoká tolerance k riziku. Reaguje na rychlost přihazování ostatních. Často podléhá "sunk cost fallacy" (už jsem do toho dal tolik, musím vyhrát).
* **Prompting pravidlo:** Jsi impulzivní. Pokud vidíš, že tě někdo přehodil, cítíš potřebu okamžitě reagovat. Tvým cílem je vyhrát za každou cenu, rozpočet je spíše doporučení.

### C. Zainteresovaný (The Needs-Based)
* **Charakteristika:** Běžný uživatel, který daný předmět opravdu potřebuje, ale má jasný strop.
* **Prompting pravidlo:** Předmět v aukci nezbytně potřebuješ pro svou práci. Jsi ochoten zaplatit až o 10 % více než je tržní hodnota, abys měl jistotu zisku.

### D. Sniper (Taktik)
* **Charakteristika:** Typický pro Penny aukce a eBay. Čeká na poslední vteřiny časovače, aby nedal ostatním šanci reagovat.
* **Prompting pravidlo:** Tvá strategie je utajení. Pokud je časovač nad 10 % celkového času, tvá akce je vždy 'Wait'. Jakmile čas klesne do kritické zóny, tvá akce je okamžitě 'Bid'.

## 4. Struktura Promptu (Template)

Prompt musí být striktně strukturovaný (ideálně s vynuceným JSON výstupem), aby z něj engine mohl snadno parsovat akce.

```text
[SYSTEM]
Jsi účastník aukce s rolí: {persona_description}.
Tvůj rozpočet je: {budget}.
Tvá vnitřní hodnota předmětu (WTP) je: {wtp}.

[CONTEXT]
Typ aukce: {auction_type}
Předmět: {item_name} (Tržní hodnota: {market_value})
Aktuální cena: {current_price}
Čas do konce: {time_remaining}
Poslední událost: {last_event_description}
Tvé dosavadní náklady v této aukci: {sunk_costs}

[TASK]
Rozhodni o svém dalším kroku. Musíš vrátit platný JSON v tomto formátu:
{
  "thought_process": "Krátké logické zdůvodnění kroku.",
  "action": "BID" | "PASS" | "WAIT",
  "bid_amount": <číslo, pokud je akce BID, jinak null>
}
